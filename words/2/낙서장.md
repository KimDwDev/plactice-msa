# pr 리뷰 답변 달기 

제 코드를 꼼꼼하게 리뷰해주셔서 감사드립니다! 🙏
남겨주신 질문 하나하나가 저의 생각을 확장하고 더 깊이 고민해볼 수 있는 좋은 계기가 되었다는 점이 정말 기쁩니다.
아래는 각 질문에 대한 저의 생각입니다.

Q: 현재 구조에서는 업로드 되는 파일 전체를 메모리 상에 올린 후에 이후 작업이 처리되기 떄문에, 대용량 파일의 경우 문제가 생길 수 있습니다. 이런 상황을 어떻게 개선할 수 있을까요?

A: 우선 Buffer.concat 사용을 최소화하기 위해 청크 단위로 처리하는 방식을 고려했습니다. (Buffer.concat은 기존 데이터를 복사하는 방식으로 이해하고 있습니다.) 다만 동영상과 같이 대용량 전송이 필요한 경우에는 TCP 기반 HTTP 대신 UDP 기반의 프로토콜이 더 적합할 수 있다는 이야기를 들은 적이 있습니다. 이 부분은 정확히는 더 공부가 필요하지만, 병렬 처리 기법과 함께 추후 꼭 연구해보고 싶은 주제입니다.

Q: auth, users 서비스를 분리한 특별한 이유가 있을까요? 일반적으로는 하나로 퉁치는 경우도 많은데, 어떤 기준으로 분리를 진행했을까요?

A: 처음에는 Auth 서비스가 로그인·로그아웃 등 보안 관련 기능을 담당하고, Users 서비스는 프로필 수정과 같은 사용자 정보 관리에 집중하도록 의도했습니다. 하지만 현재 구현에서는 Auth가 JWT 검증에만 사용되고 있어, 본래 의도와는 다소 차이가 있는 구조가 되었습니다. 이 부분은 개선 포인트로 삼아 다음에 구조를 보완하려 합니다.

Q: 현재는 Services 계층에서 DB 관련 역할까지 담당하는 것으로 보입니다. 만약 Service 부분을 Service/Repository 로 분리한다면 어떤 기준으로 분리할 수 있을까요?

A: 현재 Service 계층에서는 DB API를 직접 호출해 데이터를 조회/저장하는 역할까지 포함되어 있습니다. 제가 의도한 MSA 구조에서는 각 서비스가 자신이 담당하는 도메인에 대한 DB만 관리하도록 설계한 것이었습니다. 혹시 리뷰에서 “Service가 DB 역할까지 담당한다”고 보신 부분이 구체적으로 어느 부분인지 궁금합니다. 말씀해주신 Repository 분리는 분명 좋은 방향이라 생각하며, 도메인 로직과 DB 접근을 명확히 나누는 기준을 연구해보고 싶습니다.
