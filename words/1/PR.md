# MSA 기본 개념 학습

## 링크: https://learn.microsoft.com/ko-kr/azure/architecture/guide/architecture-styles/microservices

## 내용:

  ### 1. 마이크로서비스의 개념 

  - 마이크로서비스는 소규모 개발 팀이 작성하고 유지 관리할 수 있는 작고 독립적이며 느슨하게 결합된 구성요소를 뜻한다. 

  - 내부 구현을 다른 서비스에 숨기면서 내부에서 잘정의된 API(REST, gRPC)나 비동기(이벤트 큐)등을 이용해서 통신한다.

  - 마이크로서비스는 도메인을 중심으로 적절한 경계를 이루면서 정의된 아키텍처이다.

  - 각 서비스는 독립된 레지스트리가 존재한다.

  ### 2. 마이크로서비스 구성 요소

  - 관리를 위해서는 오케스트레이션이 존재하고 Kubernetes, Azure Container Apps이 이에 해당하고 노드 간에 서비스를 예약 및 배포하고 오류를 감지하며 실패를 복구하는 역할을 한다.

  - API 게이트웨이가 클라이언트와 남북 커뮤니케이션을 담당하고 NGINX가 게이트웨이와 통신하며 각각의 서비스가 동서 커뮤니케이션을 담당한다.

  - Apache Kafka 및 Azure Service Bus와 같은 메시징 플랫폼은 서비스간의 결합을 더욱 느슨하게 하며 비동기가 가능하게 한다.

  - 중앙 집중식 로그와 함께 모니터링 에이전트 및 OpenTelemetry와 같은 프레임워크를 사용한 실시간 모니터링을 진행해야 한다.

  - 데이터를 관리하기 위해 sql, nosql등 다양하게 사용할 수 있고 분산형 모델에 맞게 데이터 스토리지를 설계해야 한다.

  ### 3. 마이크로서비스의 혜택

  - 마이크로서비스는 각 서비스가 독립적으로 운영 배포함으로 민첩하게 확장 및 업데이트 및 구현을 할 수 있다.

  - 소규모의 포커스 있는 팀을 운영하여 집중적으로 서비스를 개선 및 운영할 수 있다.

  - 서로의 서비스가 독립적이기 때문에 작은 코드 베이스로 운영이 가능하다 모놀리식의 경우 시간이 지남에 따라 어쩔수 없이 연관이 지어질수 밖에 없는데 마이크로서비스는 이를 개선한다.

  - 다양한 기술을 사용할 수 있다. 서비스간의 독립적이다 보니 각 서비스 간의 여러 기술을 적용할 수 있다.

  - 서비스가 독립적이다 보니 오류가 서로 격리가 된다. 오류 격리로 인해서 하나의 서비스가 다운되어도 다른 서비스를 일단 운영할 수 있다.

  - 확장성이 뛰어나다 각 서비스는 독립적이어서 경계를 지킨다면 해당 서비스에서 확장성은 뛰어남을 알 수 있다.

  - 데이터 격리 하나의 마이크로 서비스는 하나의 레지스트리에 영향을 받음으로 스키마 변경이 모놀리식에 비해 자유롭다.

  ### 4. 마이크로서비스의 과제

  - 마이크로서비스는 모놀리식보다 이동 부분이 더 많아서 서비스는 단순하나 전체 시스템은 더욱 복잡하다. 애플리케이션을 디자인할때 데이터 일관성, 트랜잭션 관리와 같은 문제를 고려해야한다.

  - 마이크로서비스는 서비스 경계를 벗어난 지점에 한해서 테스트 하기가 어렵다. 

  - 마이크로서비스는 각 서비스가 독립적이다 보니 거버넌스가 부족할 수 밖에 없다 따라서 규격화된 방식을 서로 합의하고 계약해야 한다.

  - 서비스간의 API호출 등이 많아질 수 있음으로 네트워크 정체 및 대기시간이 길어질 수 있다 이를 방지하기 위해서 비동기 통신 패턴이 필요한 부분을 잘 이용해야 한다.

  - 데이터 무결성을 유지하기 힘들 수 있다 기존의 트랜잭션은 내부의 서비스에는 유용할 수 있어도 전체 서비스에는 좋지 않을 수 있다 따라서 전체 서비스에 무결성을 유지하기 위한 트랜잭션이 필요하다.

  - 마이크로서비스는 경영하기 어렵기 때문에 수준높은 DevOps 문화권이 필요하다.

  ### 5. 마이크로서비스 모범사례

  - 비즈니스 도메인을 중심으로 서비스를 모델링해야 하고 지나치게 세분하게 나누지 않도록 주의해야 한다.

  - 코드나 데이터 스키마를 공유해서는 안된다 개별 팀은 엔드 엔드 서비스를 설계하고 빌드할 책임이 있다.

  - 거버넌스를 지키기 위해 사용할 언어와 프레임워크 수를 제한하여 기술 선택을 표준화 한다.

  - 잘설계된 API를 이용해야 한다 도메인을 그대로쓰는 것이 아닌 이를 모델링 해서 전달해야 한다.

  - 서비스 간의 결합을 방지해야 한다. 공유 데이터 베이스를 사용해서는 안된다.

  - 각 서비스가 너무 수다스럽게 통신하면 안된다. 각 서비스는 높은 응집력을 가지고 있어야 하고 다른 서비스와 느슨한 결합을 이루어야 한다.

  - 비정상 상황 엔지니어링을 사용해서 마이크로서비스 아키텍처의 종속성및 복원력을 테스트 한다. 

  - 중앙 집중식 로깅, 분산 추적을 구현하여 관찰 가능성을 보장해야 한다.

  ### 6. 마이크로서비스의 안티패턴

  - 과거 또는 미래의 이벤트의 종속되게 디자인을 하면 시스템 안정싱을 감소시킨다.

  - 데이터베이스 엔터티를 이벤트로 사용하지 말고 dto payload를 사용하여 계약조건을 이벤트로 사용해야 한다.

  - 적절한 데이터 중복은 오히려 자율성이 향상되고 종속성이 사라지는 계기가 된다.

  - 내부의 서비스는 외부에 노출시키지 않고 api gateway를 이용해서 안전하게 진입하도록 하는 것이 좋다.
