# 📌 MSA 기본 개념 학습

## 📖 참고 링크: MSA 아키텍처 개요 (Microsoft Docs)

## 1. 마이크로서비스의 개념

  - 소규모 팀이 작성하고 유지할 수 있는 작고 독립적이며 느슨하게 결합된 구성 요소.

  - 내부 구현은 숨기고 API(REST, gRPC) 또는 이벤트 큐 등으로 통신.

  - 도메인 중심의 아키텍처 경계 설정.

  - 각 서비스는 독립된 레지스트리 보유.

## 2. 마이크로서비스 구성 요소

  - 오케스트레이션: Kubernetes, Azure Container Apps → 서비스 배포·오류 복구.

  - API 게이트웨이: 클라이언트와의 진입점, 남북 통신 관리(NGINX 등).

  - 메시징 플랫폼: Kafka, Azure Service Bus → 서비스 간 비동기·느슨한 결합.

   -모니터링: 중앙 로그 + OpenTelemetry 기반 실시간 모니터링.

  - 데이터 관리: SQL·NoSQL 등, 분산형 모델에 맞게 스토리지 설계.

## 3. 마이크로서비스의 장점

  - 독립적 배포/운영 → 민첩한 확장과 업데이트 가능.

  - 작은 코드 베이스 유지 → 모놀리식의 복잡성 개선.

  - 다양한 기술 스택 적용 가능.

  - 장애 격리 → 한 서비스 장애가 전체로 확산되지 않음.

  - 확장성 및 데이터 스키마 유연성 보장.

## 4. 마이크로서비스의 과제

  - 전체 시스템이 복잡해지고, 데이터 일관성·트랜잭션 관리 어려움.

  - 서비스 경계를 넘어서는 테스트가 까다로움.

  - 거버넌스 부재 가능성 → 표준화된 계약·합의 필요.

  - API 호출 증가 → 네트워크 지연 위험.

  - 데이터 무결성 유지 어려움, 분산 트랜잭션 필요.

  - 운영·관리 난이도 ↑ → 높은 수준의 DevOps 문화 요구.

## 5. 모범 사례

  - 비즈니스 도메인 중심 모델링, 과도한 세분화 금지.

  - 코드/데이터 스키마 공유 금지 → 팀별 end-to-end 책임.

  - 언어/프레임워크 표준화로 거버넌스 유지.

  - 잘 설계된 API와 DTO 기반 전달.

  - 공유 DB 금지 → 서비스 간 결합 최소화.

  - 높은 응집력 + 느슨한 결합.

  - Chaos Engineering으로 복원력 테스트.

  - 중앙 로그·분산 추적으로 관찰성 보장.

## 6. 안티패턴

  - 과거/미래 이벤트에 과도한 의존 → 안정성 저하.

  - DB 엔터티를 직접 이벤트로 사용 금지 → DTO Payload 사용.

  - 적절한 데이터 중복 허용 → 자율성 강화.

  - 내부 서비스 직접 노출 금지 → 반드시 API Gateway 경유.

## 📺 참고 링크: Pull Request 개념 (YouTube)

  - PR(Pull Request) = 여러 사용자가 코드를 병합(Merge)하기 전 검토·승인 요청하는 과정.

  -  방식:

    - Fork 기반 PR: 외부 기여자가 원본 저장소를 fork → 수정 → PR 제출.

    - Collaborator 기반 PR: 승인된 사용자들끼리 동일 저장소에서 브랜치 생성 후 PR 제출.
