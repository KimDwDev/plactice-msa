# pr 공부

- 보통 master는 실행 가능해야 한다는 문제가 있기 때문에 보통 여기서 작업을 하지는 않고 feature를 만들어서 작업을 진행한다.

- 우리가 새로운 브런치를 만들고 원격 저장소에 새롭게 push를 하면 pr이 생성이 된다. 

- assignees는 작업의 일을 할 사람을 지정하는 작업이다. 

- 코드에 대한 품평을 주었으면 하는 사람을 등록하면 작업을 할 수 있다. 

- draft pull request를 사용하게 되면 병합이 되지 않는 pr을 요청한다 이를 이용해서 리뷰어에게 리뷰를 남길 수 있도록 할 수 있다.

- conversation은 이 pr에서 일어난 행동등을 이야기 하는 것이다.

- commits는 history를 말하고 file changed는 디렉토리의 변환 상태를 말한다.

- 파일 체인지를 통해서 변환 내용을 볼 수 있고 아래에 병합을 신청할 수 있다. 

- 대화에서 conversation을 시행하게 되면 merge 작업이 실행하게 되고 이에 대해서 작업을 진행할 수 있는 것이다. 

- 이를 fetch로 받고 이에 대해서 로컬에서도 병합 작업을 하면 그대로 사용할 수 있는 것이다.

# msa 마이크로소프트 문서 보고 공부

## msa의 기본 개념

- 마이크로서비스의 장점은 확장성이 좋고 독립적으로 배포할 수 있고 신속하게 진화할 수 있는 아키텍처 스타일 이다.

- 시스템을 설계 배포 및 운영하는 방법에 대해서 학습을 해야 한다. 

- 각 서비스가 독립적이어야 하고 제한돈 컨텍스트 내에서 단일 비즈니스 기능을 구현해야 한다. 

- 각 서비스는 다른 서비스와 강하게 얽히지 않고 자기 책임 안에서만 동작해야 한다. 

- 배포 사이클, 데이터 저장소, 로직 등 다른 서비스와 상관 없이 서비스 내부에서 결정을 하면 되는 사항이다. 

- 경계를 명확히 두어서 이 서비스에서의 변경이 다른 서비스에게 피해가 가지 않도록 잘 정리해두어야 하는 사항이다. 

- 하나의 서비스는 하나의 도메인 모델을 갖고 그 모델이 의미있는 범위 까지만 책임을 져야 한다.

- 예를 들어서 전자상거래 시스템에서 주문 -> 결제 -> 배송 시스템이 있다면 주문은 주문만 하고 다른 것의 로직은 전혀 몰라야 하고 결제는 결제만 하고 주문이 어떻게 되었는지 확인하지 않아야 하며 배송도 이와 마찬가지다.

- 하나의 서비스 변경이 다른 서비스에게도 피해가 간다면 그건 잘못 경계를 설정한 것이다. 

- 경계를 잘 설정하기 위해서는 도메인 중심으로 생각을 해야 한다. 

- 특정 엔티티는 반드시 하나만 소유해야 한다.

- auth - post로 가는 것이 아니고 auth는 로그인, 회원가입, 로그아웃등 인증 로직 등을 담당하고 그 걸 해석해서 써야 하는 post, user의 경우 각각 그걸 만들어야 하고 이걸 팀에 모듈을 만들어서 같이 사용하면 되는 것이다. 

- 마이크로 서비스는 잘 작성된다면 소규모 개발자 팀이 각각의 기술을 개발할 수 있고 각 서비스는 서로 프레임워크, 라이브러리 등을 공유할 필요가 없다. 

- 전체를 빌드할 필요없이 각각의 애플리케이션을 빌드하거나 배포할 수 있는 것이다. 

- 내부 구현 은 서로 이야기 하지 않고 API즉 명확하게 계약한 내용으로만 대화를 하는 것이 목표 이다.

- 팀-서비스 1:1 매칭으로 팀이 독립적으로 기획 -> 개발 -> 배포 -> 운영까지 책임을 져야 하는 것이다.

- 비즈니스 기능 기준으로 잘라서 서비스를 운영해야 한다. 그러니까 하나의 유스케이스 단위로 로그인 -> 결제 -> 처리 -> 배송으로 유스케이스를 작성했다면 이에 따라서 각각 서비스를 작성해야 하는 것이다.

- 초기에 전략은 모노레포 + 폴더로 경계를 만든 후 스케일 시 프로세스/배포 단위로 분리해야 한다.

- 데이터 베이스는 다른 서비스의 DB를 집적 조인하지 않는 것이 원칙이다. 만약 필요한 데이터가 있다면 API 호출이나 이벤트 구독으로 가져와서 자기 로컬 읽기 모델을 만들어야 한다. 

- 일관성을 유지하기 위해서 사가(Saga) + Outbox 패턴 + 최종 일관성 이런 패턴을 사용한다고 한다 한번 자세히 알아보자. 

- OUTBOX는 STATUS 열을 만들어서 만약 읽으려고 할때 status 상태를 보고 멈추거나 실행을 시킬지 결정한다고 한다. 

- 예를 들어서 프로필 정보를 수정햇다고 치면 원천 서비스인 유저가 정상 커밋을 한 후 이 정보를 다른 서비스에서 사용하기에는 지연이 일어날 수 있다. 이때 UI를 이용해서 대처를 할 수 있다.

- API gateway를 이용해서 api를 각각 요청해서 한번에 받아서 gateway에서 내려줄 때도 있다.

- Outbox 패턴에서 kafka, nat, sqs등이 사용되는 걸로 예상이 된다. 이에 대해서 한번 학습해주어야 겠다. 

## msa 구성 요소

- msa는 유스케이스 기반에 서비스 말고도 다양한 구성요소가 존재한다.

- 마이크로서비스 서비스는 독립적으로 배포, 확장, 복구 해야 가치가 나오는건데 이때 오케스트레이션이 설계와 운영 모두 가능하게 해주는 운영 자동화 인프라이다.

- 서비스 A만 교체 하고 싶을때 일단 새로운 서비스로 교체하고 거기에 흘러보내고 문제가 생기면 자동으로 롤백을 진행한다. 

- 오케스트레이션에서 오토스케일링을 사용해서 트래픽에 따라서 자동으로 유지하는것이 중요하다.

- 헬스체크 & 자가복구를 이용해서 특정 노드 장애에도 나노 파드를 재스케줄하는것이 중요하다. 

- msa의 서비스 단위 자율성을 사람대신 플랫폼이 집행하도록 한다. 즉 배포, 확장, 복구 하는 것을 자동으로 진행해준다. 

- msa는 부분 장애를 허용하고 이에 따라서 빠른 롤백이 가능하도록 도구로 보장을 하는 것이다. 

- 쿠버네틱스랑 azure 부분의 공통점은 서비스 단위 배포/확장/복구/모니터링을 제공해서 MSA의 운영 복잡도를 낮추는 것이 주요한 목표이다.

- 쿠버네틱스는 오케스트레이터의 기능을 하며 이를 이용해서 MSA의 배포 확장, 복구 등을 관리 할 수 있다.

- posts -> comments 호출 등과 같이 다른 api 호출의 이름을 논리이름으로 하는것이 꽤 좋다. 왜냐하면 IP가 수시로 바뀌기 때문에 호출자가 IP를 모르도록 논리이름으로 가도록 변경해주는 것이 중요하다.

- IP 주소 또한 모르도록 하는 것이 결합도를 낮추는 즉 느슨하게 만드는데 도움을 준다.

- 쿠버네틱스에서는 클러스터 DNS라는 것이 존재한다 이를 이용해서 DNS로 전송이 가능하다.

- 쿠보네틱스의 경우 CONFIG/암호 시스템을 이용해서 환경변수라던지 암호 정보등을 등록할 수 있다. 

- API 게이트웨이에서 인증, 로깅, 부하 분산과 같은 횡단 관심사를 처리한다. Nginx는 내부 서비스간 통신을 지원하는 것이고 API 게이트웨이는 가장 바깥쪽에서 클라이언트의 진입점 역할을 한다. 

- 즉 AUTH만 개인적으로 시크릿 키를 쓰고 URL을 제공해서 다른 곳에서 요청을 해서 받아오는 식으로 하는 것이 매우 중요하다. 

- 항상 내부에서도 IP/포트로 사용하지말고 보통 내부 서비스 끼리도 HTTPS를 쓰는 것이 바랍직하다 이게 동서 트래픽으로 좋다.

- 게이트웨이는 북남 이고 ENVOY/프록시 같은 경우는 동서 통신을 표준화 해서 보호한다.

```scss
[모바일/웹] 
     │  (North–South)
     ▼
[ API Gateway (WAF/TLS/JWT/RateLimit/BFF) ]
     │
     ▼
[ posts ] ←(East–West)→ [ comments ] ←→ [ users ]
    ▲                         ▲               ▲
  (Envoy)                  (Envoy)         (Envoy)
   사이드카                  사이드카          사이드카
```

- 메시지 지향 미들웨어를 사용함으로서 느슨한 결합을 촉진한고 높은 확장성을 지원하며 마이크로 서비스에서 비동기 통신이 가능하게 한다.

- 관찰가능성 3요소에는 로그, 메트릭, 분산 트레이싱이 존재하고 중앙에서 수집하며 이때 trace_id를 gateway에서 주입한다. 분산 트레이싱은 호출 체인을 시간 순서로 보여주며 어디서 병목/실패가 났는지 한눈에 파악해야 한다.

- OpenTelemetry(OTel)는 이를 자동으로 수집하게 해주는 도구이다.

- 데이터 베이스를 잘 디자인 해야 하는데 이때 각 서비스는 독립적인 데이터와 스키마를 소유한다 이를 통해 서비스간 종속성을 줄이고 서비스가 독립적으로 발전할 수 있게 한다. 

## msa의 혜택

- msa는 독립적을 배포되는 덕분에 버그를 수정하고 기능 릴리스를 관리하는 것이 더 쉽다. 기존과 다르게 전체 릴리스 프로세스를 차단하는 것이 아닌 해당 부분에 대해서 수정하고 업데이트를 진행할 수 있는 것이다.

- msa는 각 서비스를 소규모의 팀이 담당하기 때문에 커뮤니케이션이 빠르고 민첩성이 높다.

- 기존의 서비스는 하나의 기능을 추가하면 다른 기능에 영향을 줄 때도 있었다 하지만 msa에서는 그러한 부분을 전혀 고려하지 않고 독립적으로 해당 서비스에 대한 오류 부분만 잘 정리하면 되는 것이다.

- msa를 사용함으로서 기존의 모놀리식 애플리케이션은 시간이 지남에 따라 코드의 연관성이 높아지는데 이 서비스는 시간이 지남에 따라 한 부분만 변경하면 됨으로 이 문제를 방지할 수 있다.

- 기술의 혼합으로서 각 서비스에 가장 적합한 기술을 선택할 수 있다.

- 오류를 격리해서 만약 하나의 서비스가 오류가 발생한다고 해서 모든 아키텍처의 서버가 다운되지는 않는다. 

- msa는 확장성에 효율적인 모습을 보인다. 즉 가장 많이 사용한다고 생각하는 부분은 더 많은 CPU 메모리를 준다거나 할 수있다.

- 데이터를 격리해서 마이크로 서비스 아키텍처는 스키마를 업데이트 하는 것이 더욱 간단하다.

-> 회로 차단 기패턴, 비동기 메시징 패턴 학습해야 한다.

## 과제들 

- 마이크로서비스는 동일한 모놀리식 애플리케이션 보다 이동 부분이 더 많이 있다. 각 서비스는 단순하지만 전체 시스템이 더 복잡하다. 서비스 검색, 데이터 일관성, 트랜잭션 관리 및 서비스간 통신과 같은 문제를 고려해야 한다.

- 주소가 동적이기 때문에 논리이름으로 찾도록 해야 하고 클러스터 DNS + Service(쿠버네틱스) / Service Registry(Consul, Eureka)등 논리이름으로 찾는다.

- 데이터 일관성을 지키기 위해서 사가, outbox, inbox등을 이용해서 중복을 방지해야 한다.

- 트랜잭션을 관리하기가 쉽지 않을 수 있다. 문제에서 댓글생성 -> 게시글 댓글수 증가 -> 알림 발송과 같이 여러 서비스가 한개의 업무 단위를 이루는 것이다. 

- 트랜잭션에 대해서 자세히 공부해봐야 할 것 같다. 

- 서비스간 통신에는 동기인 (REST, gRPC) + 비동기(이벤트)를 혼용해서 사용한다. ( 즉시성을 원할때는 동기를 결합도를 낮출때는 이벤트를 혼용해서 사용 )

- 기존 모놀리식 또는 계층화된 애플리케이션을 작성하는 것은 어렵다 즉 서비스를 넘어가는 기능에 대해서는 리펙토링이 어렵고 서비스 종속성을 테스트 하는것도 어렵다.

- 마이크로서비스는 로컬에서 전체를 띄우는 것이 어렵고 느리다. 

- 경계 때문에 리펙토링에 제약이 발생하게 된다. 즉 계약 관계가 있기때문에 계약관계가 있는 코드 같은 경우 함부로 변경할 수 는 없고 계약을 바꾸고 리펙토링을 실시해야 한다.

- Outbox + inbox에 대해서 학습을 해봐야 할 것 같다. 

- 거버넌스가 부족해서 전체 애플리케이션이 유지하기 힘들어질 수 있다. 그래서 전체 표준을 만들어서 어느정도 관리를 해야 거버넌스의 역할을 할 수 있다.

- 서비스간 종속성 체인을 신중하게 계산해야 한다. 즉 비동기 통신 패턴을 사용할 장소를 찾고 이를 잘 활용해야 한다. 

- 사가의 개념이 너무 이해가 안간다. 

- 오케스트레이션을 이용해서 오케스트레이터가 각 서비스를 명령으로 순서를 호출하고 만약 실패시 보상작업이 실시가 된다. 

- BASE 가용성 위주로 작성을 하고 잠시 불일치 허용을 통해서 일치시키는 방향으로 

- 마이크로 서비스 아키텍처에는 완성도 높은 DevOps 문화권이 필요하다 전체 서비스의 상관관계 로깅이 까다로울 수 있으나 일반적으로 로깅은 단일 사용자 작업에 대한 여러 서비스 호출을 상호 연결해야 한다.

- 마이크로 서비스는 고도로 분산된 시스템으로서 성공을 위한 기술과 경험을 가진 팀을 분배해야 한다. 

## 모범사례 

- 지나치게 서비스를 세분화 할 필요는 없다 이는 복잡성만 높이고 성능은 줄일 수 있기 때문에 명확한 서비스 경계를 가질 수 있도록 경계를 정의해야 한다. 

- 분산해서 작업해야 한다. 즉 개별 팀은 엔드 엔드 서비스를 설계하고 빌드할 책임이 있음으로 코드 또는 데이터 스키마를 공유하지 말아야 한다.

- 사용하는 언어, 프레임워크 수를 제한해서 기술 선택을 표준화 해주어야 한다. 이를 위해서 모니터링 및 배포를 위한 플랫폼 전체 표준을 설정해야 한다.

- DDD가 마이크로서비스 아키텍처와는 잘 맞는 아키텍처임을 기억해야 한다. 

- 서비스당 DB가 원칙이고 필요한건 API 호출 또는 이벤트 구독으로서 로컬 읽기 모델을 만들어야 한다. 

- 크로스 DB조인, 공용 RDS 스키마를 사용하는 것은 피해야 할 것이다.

- 왠만하면 비동기 이벤트로 느슨하게 연결하는 것이 좋다.

- 결합을 방지하는 것이 매우 중요하다. 

- 너무 과분하게 쪼갤 필요없이 쪼갤 필요가 있을때 분리하는 것이 중요하다. 

- 게이트웨이는 비즈니스 규칙을 몰라야 하고 게이트 웨이는 요청을 확인하고 전달하는 역할만 해주어야 한다. 

- 인가 정책은 각 서비스에서 담당을 해야 한다.

- 두 서비스가 너무 수다스럽게 통신하지 않아야 하고 함께 변경될 가능성이 있는 함수는 패키지 및 배포해야 한다. 

- 오류를 격리해야 한다 즉 하나의 서비스의 오류가 발생했을때 다른 서비스도 같이 영향을 받는 것이 아닌 다른 서비스는 평온하게 하도록 해주는 것이 좋다 타임아웃 등등 

- 비정상 상황 카오스에 대해서 엔지니어링하고 이에 대해서 약한 고리를 찾아야 한다. 

- 어디서 느려지는지 로그에는 trace_id, x-request-id를 항상 포함해야 한다. 이에 대해서 추적할 수 있도록 그리고 JSON을 한 줄 로그로 표현하면 된다.

- 메트릭은 RED로 요청율, 오류율을 지연하고 이에 대해서 카나리/롤백 트리거로 사용한다. 

- 분산 트레이싱은 서비스 경계를 넘는 호출 체인을 시각화 한다. 

- 항상 이에대해서 관찰하고 느려지는 지점을 빠르게 파악후 처리할 수 있어야 한다.

## 마이크로 서비스에 대한 안티패턴

- 비즈니스 도메인에 대해서 깊은 이해를 가지고 마이크로 서비스를 구현해야 한다 이 서비스 경계가 제대로 조정되어야 마이크로 서비스를 안정적으로 진행할 수 있다.

- 데이터의 진실을 담당하는 SoT를 명확하게 잡는것이 매우 중요하다. 

- SoT를 명확히 하고 함께 자주 바뀌는 규칙/데이터를 한 서비스에 묶는 것이 중요하다.

- 과거와 미래에 종속되지 않도록 이벤트를 설계해주어야 한다. 만약 하나의 이벤트에 종속된다면 그 요소를 수정하는데 종속적인 부분이 생기기 때문에 어려움으로 이에 대해서 보완을 해주어야 하는 것이다. 

- 내부 엔티티를 그대로 보내게 된다면 서비스 사이에 강결합이 발생할 수 있음으로 올바른 DTO/이벤트 스키마를 설계하여 필요한 데이터만 주로 사용하는 것이 좋다.

- 일반 이벤트를 사용하지 말고 

- 의도적인 중복을 만들어서 다른 서비스에 매번 의존하지 않는 방법도 좋은 방법중 하나이다. 

- 이러한 중복데이터를 만드는 것은 자율성을 높이는 것이다 이 데이터 베이스는 읽기용 사본으로 만들어야 한다는 것을 잊지 말아야 한다.

- 이벤트는 도메인 언어로 처리하고 명확한 이름이 필요하다. 

- 마이크로 서비스 간에 공통 라이브러리 또는 종속성을 공유하지 말고 만약 공유한다면 긴밀한 결합이 발생한다. 하나의 라이브러리의 변경으로 인해서 자체 서비스의 원칙에 어긋날 수 있기 때문이다.

- 마이크로 서비스의 외부 노출을 최소화 하고 API 게이트 웨이를 사용해서 처리하도록 한다.

- 구성값을 외부화 하는 것이 좋다 즉 환경변수로 지정해서 런타임에서 읽도록 하는 것이 좋다. 

- 무거운 보안 운영은 플랫폼으로 비즈니스 권한 판단은 서비스로 해서 도메인 내부에서 도메인 인가를 수행한다.

- 게이트웨이가 인증했다고 무조건 신뢰하는 것은 좋지 않다.

- 마이크로 서비스 자체에서 복잡한 인증로직을 다루면 너무 복잡하고 유지관리가 어려워진다.

- 보안은 플랫폼에서 가벼운것은 서비스로 나눠서 처리하는 것이 좋다.

- 즉 보안적인 부분은 오프로드 하는 것이 서비스가 집중되고 깨끗하게 관리되는데 유리하다.
